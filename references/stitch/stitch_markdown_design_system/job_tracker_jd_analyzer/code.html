warning 01 / 问题背景

Pain Point 1

海量投递中，求职者难以高效追踪多平台（猎聘、Boss、官网等）的投递状态，岗位信息极度分散。

Pain Point 2

人工分析长篇 JD 提取核心技能并修改简历耗时，且不同公司的技能要求难以量化比较，容易遗漏重点。

apps 03 / 核心功能 (Core Features)

岗位录入与解析

支持手动粘贴 JD 或从入口库导入。自动提取关键技能、生成匹配分和雷达图比对。

投递决策队列

基于解析结果和偏好配置，生成待处理队列。集中管理所有来源的投递进度。

招聘入口库

统一管理各大招聘平台的搜索链接和过滤条件，便于日常快速扫雷与录入。

全局数据看板

可视化统计投递转化率漏斗、高频技能词分布，指导简历迭代与技能提升。

memory 07 / AI Integration

使用 LLM 结合结构化输出工具（如 OpenAI JSON schema 或 Instructor），将非结构化长文本强制解析为预定义的 Pydantic 数据模型。

Prompt 核心策略：

  • 严格遵守 JSON 格式返回。
  • 从原文中提取硬性技能，丢弃软性描述（如"有责任心"）。
  • 基于预设的用户技能栈计算一个基础的 match_score (0-100)。

30秒 (电梯演讲版) expand_more
我开发了一个基于 Next.js 和 FastAPI 的智能求职管理系统。它不仅能像看板一样追踪投递状态，还能调用 LLM 自动将几千字的 JD 解析成结构化的技能要求，帮我快速比对自身匹配度。这个项目解决了我自己求职信息分散的痛点，也验证了我全栈开发和 AI 接口集成的能力。
1分钟 (核心亮点版) expand_more
Job Tracker 是我为了提高求职效率独立开发的全栈应用。痛点在于海量投递时岗位信息极度分散，且逐字阅读 JD 提炼技能非常耗时。我使用了 Next.js 做前端看板，FastAPI 加 PostgreSQL 做后端支撑。

最大的亮点是 JD Analyzer 模块：我设计了 Zero-shot Prompt，让大模型将大段 JD 文本直接输出为包含硬性要求、加分项的结构化 JSON，并生成技能词云。整个项目从写 PRD 到部署完全采用标准工程化流程，目前已帮助我高效管理了上百个投递流转。
3分钟 (STAR 原则深度版) expand_more

Situation: 在准备求职时，我发现需要在多个平台间来回切换查看进度，且手动分析每家公司 JD 的核心需求非常低效。

Task: 我决定开发一个 MVP 版本的求职看板，要求具备状态管理和自动化的 JD 解析能力，同时作为自己全栈和工程化能力的练手项目。

Action:
1. 工程规范：先写 PRD 定义边界，用 Trello 拆解前后端任务卡。
2. 架构设计：前后端分离，Next.js 负责复杂状态的看板交互，FastAPI 保证接口响应，数据存入 PostgreSQL，最后 Docker 容器化部署。
3. AI 集成：把 LLM 当作高级正则工具。我调试了特定的 Prompt，强制模型以 JSON 格式提取硬性指标和技能关键词，而非生成废话。

Result: 产出了一个稳定可用的 SaaS 雏形。它将我单次分析岗位和记录的时间从 5 分钟缩短到 30 秒内，清晰的数据大盘也极大缓解了求职焦虑。下一步计划开发浏览器插件，实现全自动化录入。
